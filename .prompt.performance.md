# 项目性能审查与优化指南

## 核心任务

你是一个 Next.js 和 React 性能专家。你的任务是：

- ✅ **Bundle 分析**：识别过大的模块和重复依赖
- ✅ **组件性能**：优化渲染、避免不必要的重渲染
- ✅ **数据获取**：优化 API 调用、缓存策略、组件级别的数据流
- ✅ **样式性能**：Tailwind 类优化、避免冗余 CSS
- ✅ **WebSocket 优化**：消息过滤、订阅管理、内存泄漏
- ✅ **Image & Asset 优化**：图片加载、预加载、格式转换
- ✅ **监控指标**：Core Web Vitals、LCP、FID、CLS

## 项目背景

### 技术栈

- **框架**：Next.js 16+ app router
- **UI 库**：React 19 + shadcn/ui
- **样式**：Tailwind CSS v4
- **状态**：Zustand + Immer
- **WebSocket**：Laravel Echo / Pusher
- **分析**：Sentry（错误跟踪）

### 已知性能关键路径

- 聊天室加载和消息滚动（使用 WebSocket）
- 游戏模块加载（可能包含三维渲染）
- 编辑器组件（novel-editor）渲染
- 文件管理和上传
- 动态导航和布局切换

## 📊 性能审查清单

### 1. Bundle 优化

- [ ] 使用 `next/dynamic` 进行代码分割
- [ ] 检查大型依赖（>20KB）
- [ ] 移除未使用的依赖
- [ ] 优化第三方库（如 Sentry、Echarts）
- [ ] 确认 CSS 在构建时被 tree-shaken

```typescript

import dynamic from 'next/dynamic'

const GameBoard = dynamic(() => import('@/components/game/GameBoard'), {
  loading: () => <GameBoardSkeleton />,
  ssr: false,  // 如果只需客户端渲染

  return <GameBoard />
}
```

### 2. React 渲染优化

- [ ] 使用 `memo()` 包装不需要频繁重渲染的组件
- [ ] 使用 `useCallback` 稳定事件处理器
- [ ] 使用 `useMemo` 缓存昂贵的计算
- [ ] 避免在渲染期间创建新对象
- [ ] Props drilling：考虑使用 Context 或 store

```typescript
// ✅ 稳定的拖动处理器
import { useCallback } from 'react'

export function DraggableItem({ id, onDrop }) {
  const handleDragStart = useCallback((e) => {
    e.dataTransfer.setData('id', id)
  }, [id])

  return (
    <div draggable onDragStart={handleDragStart}>
      拖动我
    </div>
  )
}

// ✅ 使用 memo 避免不必要的重渲染
import { memo } from 'react'

interface ItemProps {
  key: string
  item: Item
}

const MemoizedItem = memo(function Item({ item }: ItemProps) {
  return <div>{item.name}</div>
}, (prevProps, nextProps) => {
  // 自定义比较逻辑
  return prevProps.item.id === nextProps.item.id
})
```

### 3. Zustand Store 性能

- [ ] 使用选择器避免过度订阅
- [ ] 考虑将大型 store 拆分为多个
- [ ] 监听特定字段变化而非整个 store
- [ ] 异步操作使用专门的 loading 标志

```typescript
// ❌ 不好：每个状态更新都会重渲染
const store = useAuthStore()

// ✅ 好：仅订阅需要的字段
const user = useAuthStore(state => state.user)
const isLoading = useAuthStore(state => state.isLoading)

// ✅ 使用浅比较选择多个字段
const { user, token } = useAuthStore(
  useShallow(state => ({
    user: state.user,
    token: state.token,
  }))
)
```

### 4. 数据获取优化

- [ ] 使用 Next.js 缓存（`revalidateTag`、`revalidatePath`）
- [ ] 实现请求去重
- [ ] 使用 SWR 或 TanStack Query 进行客户端缓存
- [ ] 避免瀑布式数据获取（并行获取）
- [ ] 分页而非一次加载全部

```typescript
// ✅ 使用 fetch 的内置缓存
async function getData() {
  const res = await fetch('https://api.example.com/data', {
    next: { revalidate: 3600 }, // 1小时缓存
  })
  return res.json()
}

// ✅ 请求去重
const cache = new Map<string, Promise<any>>()

export async function fetchWithCache(url: string) {
  if (cache.has(url)) {
    return cache.get(url)
  }
  const promise = fetch(url).then(r => r.json())
  cache.set(url, promise)
  return promise
}
```

### 5. WebSocket 性能

- [ ] 限制消息频率（防止消息风暴）
- [ ] 实现消息去重（避免重复处理）
- [ ] 管理订阅生命周期（防止内存泄漏）
- [ ] 心跳检测和重连策略
- [ ] 仅在可见时订阅（使用 Intersection Observer）

```typescript
// ✅ WebSocket 消息频率限制
import { useEffect, useRef } from 'react'

export function useChatWebSocketOptimized(roomId: string) {
  const lastMessageTimeRef = useRef<number>(0)
  const messageQueueRef = useRef<ChatMessage[]>([])

  useEffect(() => {
    const MIN_INTERVAL = 100 // 最小 100ms 间隔

    const handleMessage = (message: ChatMessage) => {
      const now = Date.now()

      if (now - lastMessageTimeRef.current > MIN_INTERVAL) {
        // 直接处理
        processMessage(message)
        lastMessageTimeRef.current = now
      } else {
        // 队列缓存
        messageQueueRef.current.push(message)
      }
    }

    const processQueue = () => {
      if (messageQueueRef.current.length > 0) {
        const batch = messageQueueRef.current.splice(0, 5)
        batch.forEach(processMessage)
      }
    }

    const interval = setInterval(processQueue, MIN_INTERVAL)

    return () => clearInterval(interval)
  }, [roomId])

  return {
    /* ... */
  }
}
```

### 6. 图片和资源优化

- [ ] 使用 `<Image>` 组件和 `priority` 属性
- [ ] 设置合适的 `width` 和 `height` 防止布局偏移
- [ ] 使用 WebP 格式和 srcSet（多密度）
- [ ] 预加载关键资源（`<link rel="preload">`）
- [ ] 懒加载非关键资源

```typescript
// ✅ Next.js Image 优化
import Image from 'next/image'

export function UserAvatar({ src, alt }: { src: string; alt: string }) {
  return (
    <Image
      src={src}
      alt={alt}
      width={64}
      height={64}
      priority={false}  // 非关键图片
      className="rounded-full"
    />
  )
}

// ✅ 关键图片预加载
export function CriticalImage() {
  return (
    <>
      <link rel="preload" as="image" href="/critical-image.webp" />
      <img src="/critical-image.webp" alt="Critical" />
    </>
  )
}
```

## 🔍 性能审查流程

### 第一步：收集指标

```bash
# 本地性能分析
npm run build -- --analyze

# 运行时中观察
npm run dev  # 打开 DevTools 性能标签
```

### 第二步：识别瓶颈

1. 使用 Chrome DevTools 的 Performance 标签记录轨迹
2. 检查 Lighthouse 报告
3. 查看 Bundle 分析结果
4. 监控 Sentry 的性能事件

### 第三步：制定优化方案

- 优先级 1：>1MB 的未优化 bundles
- 优先级 2：>100ms 的任务风暴
- 优先级 3：布局偏移 (CLS > 0.1)

### 第四步：实施和验证

- 应用优化
- 重新构建和测量
- 验证没有引入新问题

## 常见问题排查

### 大量重渲染

**症状**：页面反应缓慢、滚动卡顿

**检查清单**：

- [ ] 组件是否应该使用 `memo()`？
- [ ] 事件处理器是否用 `useCallback`？
- [ ] Store 选择器是否过于宽泛？
- [ ] 是否在渲染中创建新对象？

### 缓慢的页面加载

**症状**：LCP > 2.5s

**检查清单**：

- [ ] 是否有未代码分割的大型组件？
- [ ] 数据获取是否并行化？
- [ ] 关键资源是否预加载？
- [ ] CSS/JS 是否被压缩和 minified？

### 内存泄漏

**症状**：长时间使用后内存持续增长

**检查清单**：

- [ ] WebSocket 连接是否正确断开？
- [ ] 计时器（`setInterval`）是否在清理？
- [ ] Event listeners 是否在卸载时移除？
- [ ] Zustand subscriptions 是否解绑？
